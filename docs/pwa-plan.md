# FamilyCut PWA 规划文档

状态：已确认方向，待按迭代实施  
更新时间：2026-04-02 UTC

## 1. 目标

本项目将继续保持单仓结构，在现有 `backend/`、`admin-web/`、`android-app/` 之外新增 `mobile-web/`，作为面向 iPhone 用户的 PWA 客户端。

本次规划的核心目标只有 3 个：

- 让 iPhone 用户尽快获得可安装、可登录、可完成核心记录的用户端。
- 尽量复用现有服务端接口和业务模型，不重写后台体系。
- 接受 PWA 与 Android 原生体验存在边界差异，但优先保证核心流程能直接落地。

## 2. 已确认产品边界

### 2.1 一定要做

- `mobile-web/` 作为同仓新前端。
- 登录、邀请加入、首次资料补全。
- 今日概览首页。
- 晨重记录。
- 手动餐食记录。
- 两种拍照识别入口：
  - 拍营养成分表。
  - 拍现成饭菜。
- 上传到服务端后由 AI 识别，再回填表单。
- 日报、周报、月报查看。
- 报表分享。
- 轻量离线缓存。
- 预留 iOS Web Push。
- 预留 Android App 原生消息推送。

### 2.2 明确不做

- 不拆成两个独立仓库。
- 不改 `admin-web/` 的后台定位。
- 不做后台持续任务。
- 不做后台自动补传队列。
- 不做重型离线同步和冲突解决。
- 不在一期实现 Web Push 或 Android Push 的完整闭环。
- 不做自拍对比功能。
- 不暴露“家庭流分享开关”或“家庭流分享入口”。

### 2.3 自拍相关统一规则

- Android App 和未来 PWA 都不再暴露“家庭流分享开关”。
- 自拍对比功能隐藏，但代码先不删除。
- 如果未来 PWA 保留自拍上传，也只保留私有记录方向，不包装成“流”。
- `admin-web` 仍可保留后台查看共享自拍数据的能力，但界面文案不再强调“流”。

## 3. 单仓组织方式

推荐目录如下：

- `backend/`
- `admin-web/`
- `android-app/`
- `mobile-web/`
- `docs/pwa-plan.md`

执行规则如下：

- 后续用户端需求默认同时评估 `android-app/` 和 `mobile-web/` 的影响面。
- 后台管理需求只落 `admin-web/`。
- 某项能力如果属于原生独有，必须在需求单或任务说明里明确写出“仅 Android”。

## 4. 技术方案

`mobile-web/` 默认采用与现有 Web 端一致的技术路线，降低维护成本：

- `Vite`
- `React`
- `TypeScript`
- `react-router`
- `@tanstack/react-query`
- `zod`
- `vite-plugin-pwa`

原因：

- 与现有 `admin-web/` 的构建体系一致，接入成本低。
- 适合快速迭代移动端页面。
- PWA 的 `manifest`、`service worker`、安装体验已有成熟工具链。

## 5. 部署与域名原则

- 优先使用 HTTPS。
- 优先与现有服务端同域部署，减少 CORS、鉴权和分享链路复杂度。
- 一期不把“可编辑服务端地址”当成 PWA 必需能力。
- 如果后续确实需要多环境切换，再单独设计开发/测试入口，不在一期设置页开放给普通用户。

## 6. 一期页面范围

建议一期只做以下页面，不发散：

### 6.1 登录与加入

页面建议：

- `/login`
- `/join/:code`
- `/profile/setup`

能力要求：

- 用户名密码登录。
- 邀请码预览和邀请码注册。
- 首次资料补全。
- 登录态恢复。
- token 失效后回到登录页。

### 6.2 首页

页面建议：

- `/home`

能力要求：

- 今日体重摘要。
- 今日餐食摘要。
- 今日热量差或报表摘要。
- 快捷入口跳转到晨重、记餐、拍照识别、报表。

### 6.3 晨重

页面建议：

- `/weight`

能力要求：

- 录入体重。
- 录入体脂。
- 查看最近 7 到 30 条记录。

### 6.4 餐食

页面建议：

- `/meals`

能力要求：

- 手动记餐。
- 最近餐食列表。
- 复制历史餐食。
- 识别结果回填后再确认保存。

### 6.5 拍照识别

页面建议：

- 合并在 `/meals` 页面内，不单独拆复杂相机模块。

能力要求：

- 拍营养成分表。
- 拍现成饭菜。
- 上传图片。
- 轮询识别结果。
- 失败可重试。

### 6.6 报表

页面建议：

- `/reports`
- `/reports/daily/:date`
- `/reports/weekly/:startDate`
- `/reports/monthly/:yearMonth`

能力要求：

- 查看日报、周报、月报。
- 展示长图。
- 导出或分享。

### 6.7 设置

页面建议：

- `/settings`

能力要求：

- 退出登录。
- 基础偏好设置。
- 不在一期加入复杂服务器配置切换。

## 7. 接口复用矩阵

### 7.1 认证与资料

| 场景 | 接口 | 用途 |
| --- | --- | --- |
| 登录 | `POST /auth/login` | 用户登录 |
| 邀请码预览 | `GET /auth/invite-preview/{code}` | 展示邀请码角色与可用性 |
| 邀请注册 | `POST /auth/register-by-invite` | 首次创建用户 |
| 当前用户资料 | `GET /members/me` | 拉取当前用户资料 |
| 更新资料 | `PUT /members/me` | 首次资料补全和后续修改 |

### 7.2 晨重

| 场景 | 接口 | 用途 |
| --- | --- | --- |
| 新增晨重 | `POST /measurements` | 提交体重和体脂 |
| 最近晨重 | `GET /measurements` | 显示最近记录 |

### 7.3 餐食与 AI 识别

| 场景 | 接口 | 用途 |
| --- | --- | --- |
| 创建识别草稿 | `POST /nutrition/drafts` | 上传营养表或食物图片 |
| 查询识别草稿 | `GET /nutrition/drafts/{draft_id}` | 轮询识别状态 |
| 提交餐食 | `POST /meals` | 用户确认后落单 |
| 最近餐食 | `GET /meals` | 历史列表、复制历史 |

### 7.4 报表

| 场景 | 接口 | 用途 |
| --- | --- | --- |
| 日报 | `GET /reports/daily/{report_date}` | 查看或触发生成日报 |
| 周报 | `GET /reports/weekly/{start_date}` | 查看周报 |
| 月报 | `GET /reports/monthly/{year_month}` | 查看月报 |
| 最近报表 | `GET /reports/recent` | 首页或报表页摘要 |

## 8. 首页数据策略

一期默认不强行新增聚合接口，优先复用现有用户态接口组合出首页：

- `GET /members/me`
- `GET /measurements`
- `GET /meals?target_date=<today: YYYY-MM-DD>`
- `GET /reports/daily/<today: YYYY-MM-DD>` 或 `GET /reports/recent`

只有在以下情况同时出现时，才新增用户端聚合接口，例如 `GET /mobile/dashboard`：

- 首屏请求数明显过多。
- 移动网络下首屏耗时明显偏长。
- 前端组合逻辑开始变得难维护。

## 9. AI 识别链路

这是一期必须直接落地的核心链路。

### 9.1 营养成分表识别

流程：

1. 用户点击“拍营养成分表”。
2. PWA 使用 `<input type="file" accept="image/*" capture="environment">` 拉起系统相机或相册。
3. 前端在本地做轻量压缩。
4. 前端调用 `POST /nutrition/drafts`，`draft_type=label`。
5. 前端拿到 `draft_id` 后轮询 `GET /nutrition/drafts/{draft_id}`。
6. 服务端返回识别结果后，自动回填：
   - 食物名称
   - 每 100g 热量
   - 碳水
   - 脂肪
   - 蛋白质
   - 钠
7. 用户确认后调用 `POST /meals`。

### 9.2 现成饭菜识别

流程与上面一致，差异在于：

- `draft_type=dish_estimate`
- 允许补充描述文本
- 服务端返回估算营养值后，仍要求用户人工确认

### 9.3 前端交互要求

- 上传中要有明确状态。
- 识别中要有轮询状态。
- 识别失败可重试。
- 页面刷新后，如果本地还存着未完成 `draft_id`，应继续查询一次最终状态。

## 10. 轻离线缓存策略

本项目需要的是“轻离线”，不是完整离线业务系统。

### 10.1 要做

- 缓存静态资源，让 PWA 壳可以离线打开。
- 缓存最近一次首页数据。
- 缓存最近一次查看成功的报表摘要。
- 缓存未提交的晨重表单。
- 缓存未提交的餐食表单。
- 缓存识别中的 `draft_id` 和本地状态。

### 10.2 不做

- 后台持续轮询。
- 后台自动补传。
- 长时间离线任务队列。
- 冲突自动合并。
- 大规模本地媒体管理。

### 10.3 建议存储分层

- `Cache Storage`：静态资源、PWA 壳、最近成功的报表图片请求。
- `IndexedDB`：晨重草稿、餐食草稿、识别草稿状态、最近首页摘要。
- `localStorage`：少量轻量偏好项，例如引导完成状态。

## 11. 认证策略

当前后端登录返回 `access_token` 和 `refresh_token`，但仓库内尚未看到公开的 refresh 接口。

一期建议：

- 登录后持久化 token。
- 应用启动时恢复登录态。
- access token 失效时直接回到登录页。

后续如果要做更平滑的浏览器会话体验，再补 refresh 接口和静默续期流程。

## 12. 分享策略

报表分享是一期刚需。

执行顺序：

1. 优先尝试 Web Share API。
2. 如果浏览器不支持分享文件或图片，则回退到下载报表图片。
3. 长图查看页面要明确提供“分享”和“下载”两个动作，不依赖浏览器默认菜单。

## 13. 推送预留

### 13.1 iOS PWA Web Push

本阶段只做预留，不做正式推送闭环。

已确认前提：

- 站点必须是 HTTPS。
- 用户需要把 PWA 添加到主屏幕。
- 服务端后续需要支持标准 Web Push 订阅和消息下发。
- 服务端需要能出站访问 `*.push.apple.com`，通常是 `443`。
- 不需要 App Store 上架。
- 不需要固定公网 IPv4。
- 不需要额外向苹果付费。
- 不需要为 APNs 单独开放入站端口。

后续真正实现时，需要补：

- Web Push 订阅保存接口。
- 订阅失效清理机制。
- 推送内容模板。
- 用户端通知开关。

### 13.2 Android App 原生推送

本阶段同样只做预留，不直接实现。

预留要求：

- 后端通知能力不要只绑定 PWA。
- 推送消息模型要能同时适配 Android App 和 PWA。
- Android 使用哪个消息通道后续单独决策，不阻塞 PWA 一期。

## 14. Android 与 PWA 的边界差异

在当前收敛范围下，PWA 与 Android 的差距已经明显缩小，但仍然存在以下边界：

- 相机体验：PWA 能拍照，但 iPhone 上系统相机唤起和预览控制通常不如原生顺。
- 系统分享链：PWA 可分享，但稳定性和一致性通常弱于原生。
- 推送门槛：PWA Web Push 依赖 HTTPS、主屏幕安装和 Apple 推送链路，原生更直接。
- 导航与流畅度：长表单、长列表、拍照切页时，PWA 更容易保留“网页感”。

但以下能力应视为 PWA 一期可直接承接：

- 登录和邀请加入。
- 晨重记录。
- 手动餐食。
- 拍照上传给服务端 AI 识别。
- 日报、周报、月报查看。
- 报表分享。

## 15. 服务端改动策略

一期原则是“尽量不动，必要时补最少”。

### 15.1 默认不动

- 认证主流程
- 晨重记录接口
- 餐食与识别草稿接口
- 报表接口

### 15.2 可选最小补充

- 如果首页接口组合导致前端复杂度过高，可补 `GET /mobile/dashboard`。
- 如果浏览器长期会话体验要求提高，可补 refresh 接口。
- 如果后续做 Web Push，再补订阅与下发接口。

## 16. 实施顺序

建议按以下顺序推进：

1. 创建 `mobile-web/` 脚手架，接好 PWA 壳、路由、鉴权骨架。
2. 完成登录、邀请加入、首次资料补全。
3. 完成首页和晨重。
4. 完成手动餐食和最近餐食。
5. 完成拍营养成分表、拍食物两条 AI 识别链路。
6. 完成日报、周报、月报查看和分享。
7. 完成轻离线缓存。
8. 最后统一收口 Android 与 PWA 的自拍边界文案和入口。

## 17. 一期验收标准

满足以下条件，视为 PWA 一期达标：

- iPhone Safari 可“添加到主屏幕”。
- 主屏幕启动后可进入 `standalone` 体验。
- 用户可以登录并恢复登录态。
- 用户可以录入晨重并看到最近记录。
- 用户可以手动记餐并复制历史餐食。
- 用户可以拍营养成分表并完成 AI 识别落单。
- 用户可以拍现成饭菜并完成 AI 识别落单。
- 用户可以查看日报、周报、月报。
- 用户可以分享或下载报表长图。
- 弱网或刷新后，未提交表单和识别中的本地状态不会直接丢失。

## 18. 后续变更规则

从本规划开始，后续用户端需求默认按以下规则讨论：

- 先判断是“Android only”、“PWA only”还是“两端都做”。
- 如果没有特别说明，默认按“两端都评估”处理。
- 任何涉及自拍、分享、推送、离线、拍照识别、报表的重大边界变更，都要同时更新本文件和交接文档。
